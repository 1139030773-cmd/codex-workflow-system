# 调试修复

只修一个问题，先找根因，再做最小修复。

## 工作流程

1. 明确现象
2. 复现问题
3. 收集相关文件和日志
4. 找根因
5. 做最小修复
6. 验证修复
7. 汇报剩余风险

## 开始前问题

每次最多问 3 个：
- 你看到的错误是什么？
- 你期望它怎样？
- 这个问题什么时候出现？

如果能从文件或命令中自行发现，不要反复问用户。

## 修复原则

- 优先最小改动
- 不顺手重构
- 不新增功能
- 不改架构，除非进入决策闸门
- 能加回归测试时，优先加小测试

## 输出格式

- 问题现象
- 根因
- 修复内容
- 验证结果
- 剩余风险

## 系统关联

如果调试中发现项目结构混乱，建议 `/drift-auditor`。修复完成后如需收尾，建议 `/phase-closeout`。
